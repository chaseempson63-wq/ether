2. System Breakdown

2.1 External Services

Supabase Auth

Email/password authentication. Client-side SDK handles login/register. Server verifies tokens via supabaseAdmin.auth.getUser(token). Auto-provisions local user row on first login.

Config

VITE_SUPABASE_URL + VITE_SUPABASE_ANON_KEY

PostgreSQL (Supabase Pooler)

13 tables via Drizzle ORM. pgvector extension for 1024-dimensional HNSW-indexed embeddings. Session-mode pooler on port 5432.

Config

DATABASE_URL (pooler connection string, prepare: false)

Venice AI — Chat

llama-3.3-70b via OpenAI-compatible API. Used for persona responses, entity extraction, follow-up questions, title generation. venice_parameters.include_venice_system_prompt: false on every call.

Config

VENICE_API_KEY → https://api.venice.ai/api/v1/chat/completions

Venice AI — Embeddings

BGE-M3 model, 1024 dimensions. Used to embed memory nodes and user queries for vector search.

Config

VENICE_API_KEY → https://api.venice.ai/api/v1/embeddings

3. End-to-End Flows

3.1 New User Registration → First Memory

Browser: /register → supabase.auth.signUp(email, password) → Supabase creates auth user → Session token returned → Redirect to /
User navigates to /reflection → tRPC call with Bearer token → createContext: supabaseAdmin.auth.getUser(token) → db.ensureUser() auto-provisions local row → ctx.user set
User types memory + clicks save → memory.create mutation → INSERT memory_nodes → processContent() fire-and-forget

Background pipeline (3-10s): Venice LLM extracts entities → fuzzy-match/create nodes → Venice LLM proposes edges → create memory_edges → Venice Embed generates vectors → update memory_nodes.embedding

3.2 Persona Chat (Graph-Aware RAG)

User types "what did my dad teach me?" → persona.chat mutation

Parallel:

buildPersonaSystemPrompt() → Fetch value + reasoning nodes from DB → Build system prompt with core values + reasoning patterns
retrieveRelevantContext() → Venice Embed (query) → pgvector top-5 → 2-hop BFS on edges → Dedup + rank (sim*0.6 + edge*0.3 + recency*0.1) → Top 15 grouped by halliday_layer
System prompt + context + conversation history + query → Venice LLM (llama-3.3-70b) → Response + truthfulness tag + source memories
Background: processContent(user message) → Entity extraction from what the user said

3.3 Halliday Interview Flow

/halliday loads → getProgress (weighted 5-category breakdown) → getNextQuestion (adaptive: weakest category first)
User answers question → Option A: "Continue" → generateFollowUp (Venice LLM) → follow-up displayed
User submits final answer → submitResponse → Calculate specificity → INSERT halliday_responses → INSERT memories (legacy) → UPDATE halliday_progress → processContent() (background)

3.4 Authentication Failure Flow

Token expired / invalid → supabaseAdmin.auth.getUser() fails → ctx.user = null → protectedProcedure throws UNAUTHORIZED
Client receives UNAUTHORIZED → Cache observer catches it → supabase.auth.signOut() → Redirect to /login (skipped if already there)

4. Gaps, Risks & Inefficiencies

4.1 Risks (will break things)

No client-side route guards

All routes (including /dashboard, /chat, /halliday) are accessible to unauthenticated users. They see broken states until tRPC calls fail with UNAUTHORIZED, then get redirected. Poor UX — shows layout with empty/errored content before bouncing to /login.

Fix: Wrap protected routes in an <AuthGuard> component that checks useAuth().isAuthenticated before rendering.

system.notifyOwner will crash at runtime

The admin procedure system.notifyOwner imports from notification.ts which references ENV.forgeApiUrl and ENV.forgeApiKey. These properties don't exist in the ENV object. Any call = instant crash.

Fix: Delete or disable the procedure until a replacement notification service is wired up.

Fire-and-forget pipeline has no error recovery

processContent() is called without await and has no retry logic. If Venice is rate-limited, down, or returns malformed JSON, the pipeline silently fails. Entities don't get extracted, embeddings don't get generated, and the graph stays incomplete — with no way to detect or recover.

Fix: Add a pipeline_jobs table with status tracking. Queue failed jobs for retry. Surface pipeline health in the dashboard.

No rate limiting on AI endpoints

Every persona.chat call triggers 1 embedding + 1 LLM call (sync) + 3 more AI calls (background). A user spamming chat burns Venice API credits with no throttle.

Fix: Add per-user rate limiting (e.g., 20 chat messages/minute, 100/day).

4.2 Gaps (missing functionality)

No beneficiary access to persona chat

Beneficiaries can be created with access levels (full/restricted/legacy_only), but there's no route, auth flow, or tRPC procedure that lets a beneficiary log in and chat with the persona. The beneficiary.getAccessibleMemories query exists but reads from the legacy memories table, not memory_nodes.

No password reset flow

Supabase Auth supports password reset via email, but there's no "Forgot password?" link or reset page in the client.

No data export

Users can't export their memories, graph, or interview responses. Critical for a "Digital Mind" product — users need to own their data.

No voice synthesis

CLAUDE.md references ElevenLabs API for voice, but there's zero integration. The persona responds only in text.

No monitoring or observability

No logging service, no error tracking (Sentry etc.), no API latency monitoring. Venice API failures are console.error only.

4.3 Inefficiencies (slow you down)

13 dead files still in codebase

Legacy Manus/Forge files that are imported nowhere and reference non-existent ENV vars:

  • server/_core/dataApi.ts
  • server/_core/imageGeneration.ts
  • server/_core/map.ts
  • server/_core/voiceTranscription.ts
  • server/_core/notification.ts
  • server/storage.ts
  • server/vectorSearch.ts (replaced by pgvector in db.ts)
  • client/src/components/Map.tsx
  • client/src/pages/ComponentShowcase.tsx

Also 4 unused npm packages: @aws-sdk/client-s3, @aws-sdk/s3-request-presigner, axios, jose.

Legacy tables still in schema

memories, reasoning_patterns, core_values are superseded by memory_nodes but still in the schema. halliday.submitResponse dual-writes to both memories AND memory_nodes (via processContent). The beneficiary.getAccessibleMemories reads from the old memories table which gets fewer and fewer writes.

Fix: Migrate beneficiary queries to memory_nodes. Stop dual-writing. Drop legacy tables after migration.

searchMemoryNodesByName does in-memory filtering

db.searchMemoryNodesByName() fetches ALL user nodes then filters with .filter(n => name.includes(...)) in JS. Should be a SQL ILIKE query.

generateEmbeddingsBatch is never called

The batch embedding function exists in embeddingService.ts but graphPipeline.embedNodes() calls generateEmbedding() in a serial loop. Should use batch API to reduce round-trips from N calls to 1.

ENV object is inconsistent

server/_core/env.ts defines an ENV object, but DATABASE_URL is read directly from process.env in db.ts, and VITE_SUPABASE_* vars are read directly in supabase.ts. Three different patterns for env access.

6. Build & Deploy

Development

pnpm dev

Single process: tsx runs server/_core/index.ts with Vite middleware. Hot reload on both client and server. Port 3000.

Production Build

pnpm build

1. vite build → dist/public/ (client assets)
2. esbuild server/_core/index.ts → dist/index.js (server bundle, ESM, external deps)

Production Start

pnpm start

NODE_ENV=production node dist/index.js
Express serves dist/public/ as static files with SPA fallback.

Railway Deployment

Nixpacks auto-detects Node.js + pnpm from package.json. Runs build → start automatically. Set all 4 required env vars in Railway dashboard.
